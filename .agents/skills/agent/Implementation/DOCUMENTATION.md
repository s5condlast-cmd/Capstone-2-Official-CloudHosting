# Documentation Guidelines

## Purpose

Keep docs and comments truthful and useful, without over-documenting.

---

# Update Docs When

- A new public API, component, or hook is added.
- Existing documented behavior changes.
- A README-level setup/usage step changes (new env var, new script, new dependency).

---

# Comments

- Explain *why*, not *what* -- the code already shows what it does.
- Remove or update comments that describe code that no longer exists after an edit.

---

# Skip Documentation When

The change is internal-only and doesn't change any public contract or usage step.
